# oxideav-audiotoolbox

macOS AudioToolbox hardware decode/encode bridge for the [oxideav](https://github.com/OxideAV/oxideav) framework.

## Why a bridge crate?

Apple's [AudioToolbox](https://developer.apple.com/documentation/audiotoolbox) exposes the dedicated audio codec engine on Apple Silicon (and equivalent paths on Intel Macs). For AAC encode/decode this is the canonical "hardware AAC" path historically credited with iPhone-quality encodes at very low CPU cost.

This crate is a **thin runtime-loaded bridge** — no compile-time link dependency on AudioToolbox, no Objective-C / Swift. The framework is opened via [`libloading`] on first use.

## Fallback behaviour

Two distinct failure paths fall back automatically to the pure-Rust codec:

1. **Load failure** — older macOS, missing framework, sandboxed environment without AT entitlements. `register()` logs and returns without registering, so the SW codec is the only candidate at dispatch.
2. **Init failure** — `AudioConverterNew` returns a non-zero `OSStatus` for the requested ASBD. Common triggers: unsupported sample rate / channel layout, encoder bitrate the device doesn't accelerate, hardware codec slot busy (concurrent-converter cap on iOS-class hardware). The factory returns `Err`; the registry retries the next-priority impl (typically the SW one).

Pipelines that require hardware can opt out of the SW fallback by setting `CodecPreferences { require_hardware: true, .. }` — the registry will then surface the `OSStatus` error instead of degrading silently.

## Platform gating

The whole crate is `#![cfg(target_os = "macos")]`. On Linux / Windows it compiles to an empty rlib; the umbrella `oxideav` crate gates the `register` call behind the same cfg.

## Priority

Hardware factories register with `priority = 10` — **lower numbers win at resolution time**, so on macOS the AudioToolbox path is preferred over the pure-Rust implementation (default priority 100).

Round 2 SNR measurement: encode → decode 440 Hz sine at 48 kHz / stereo / 128 kbit/s → **36.7 dB** per channel (well above 25 dB threshold).

Round 3 ALAC round-trip: encode → decode a 2-second sine+LCG-noise mix at 48 kHz / 16-bit stereo, **190,464 / 192,000 samples bit-exact** (zero priming silence on the AT path) — proves the encoder's vended magic cookie + decoder property wiring is correctly wired and the codec is truly lossless end-to-end.

Round 4 HE-AAC round-trip: encode → decode a 2-second 1 kHz sine at 48 kHz / stereo, **HE-v1 @ 64 kbit/s ≈ 11 dB SNR, HE-v2 @ 32 kbit/s ≈ 10 dB SNR** per channel — SBR's patch-and-scale upper-band reconstruction caps recoverable phase fidelity well below transparency, so the test asserts only that the pipeline is wired correctly (the framework's encoder quality is not under our control). Profile selection is via `CodecParameters::options.insert("profile", "he" | "he-v2")`. HE encoder publishes a 42-byte ISO/IEC 14496-1 esds descriptor (AOT extension) through `output_params.extradata`; the matching decoder consumes it via `kAudioConverterDecompressionMagicCookie` to bypass the `kAudioCodecBadDataError` (`'bada'`) rejection that plain AAC LC config triggers on HE bitstreams.

Round 5 AAC-LD / AAC-ELD round-trip: encode → decode a 2-second 1 kHz sine at 48 kHz / stereo @ 128 kbit/s, **AAC-LD ≈ 29 dB SNR, AAC-ELD ≈ 26 dB SNR** per channel. These are the conferencing-oriented low-delay AOTs (`kAudioFormatMPEG4AAC_LD` = `'aacl'`, AOT 23; `kAudioFormatMPEG4AAC_ELD` = `'aace'`, AOT 39): the shortened analysis/synthesis window cuts algorithmic delay to ~15-20 ms (against AAC LC's ~100+ ms) — the win is latency, not compression, so they run at full-band LC-class bitrates and reach near-transparent SNR. AudioConverter packetises both at **512 PCM frames per packet** with **no SBR upsample** at the converter boundary (output rate = input rate). Selected via `CodecParameters::options.insert("profile", "ld" | "eld")`. Neither has an ADTS representation (ADTS profile bits encode only Main/LC/SSR/LTP), so — like HE — packets are emitted as raw AAC bytes and the AOT travels out-of-band in the encoder-vended magic cookie (`output_params.extradata`), consumed on decode via `kAudioConverterDecompressionMagicCookie`.

Round 7 AMR-NB (Adaptive Multi-Rate Narrowband, 3GPP TS 26.071 / RFC 4867) **decode**: a 20 ms / 8 kHz mono narrow-band speech codec with **8 speech modes** (MR475 4.75 kbit/s → MR122 12.2 kbit/s), plus an SID (silence-descriptor) frame and a NO_DATA (DTX-skip) marker. AudioToolbox exposes AMR-NB through `kAudioFormatAMR` (`'samr'`) as a **decode-only** target — there is no paired AT encoder, so the registry only installs an `amr_nb` decoder factory. Storage-format packets per RFC 4867 §5.1: one **TOC byte** (`F FFFF Q PP`) carrying frame-type + follow-up + quality bits, followed by the per-mode compressed payload (1, 6, 13, 14, 16, 18, 20, 21, 27 or 32 total bytes including the TOC). The decoder validates each incoming packet against the TOC-derived size table before queueing, rejecting both reserved frame types (FT 9..=14) and size mismatches with `Error::invalid`. Empirical AT behaviour: the decoder vends PCM in **120-sample S16 mono blocks** (15 ms at 8 kHz) per `FillComplexBuffer` call rather than the 160-sample analysis frame size — an internal AT chunking detail that the per-frame size invariant (`af.samples × 2 == af.data[0].len()`) handles transparently. Persistent input-queue + one-packet-of-slack lookahead pattern (matching iLBC / HE-AAC) so AT never sees "0 packets" mid-stream. Tags claimed: FourCC `'samr'` (3GPP / ISOBMFF) + Matroska `A_AMR/NB`. No magic cookie — the per-packet TOC byte carries everything AT needs.

Round 8 AMR-WB (Adaptive Multi-Rate Wideband, 3GPP TS 26.171 / 26.201 / RFC 4867) **decode**: a 20 ms / 16 kHz mono wideband speech codec with **9 speech modes** (MR660 6.60 kbit/s → MR2385 23.85 kbit/s), plus an SID (silence-descriptor) frame at FT=9 — one slot higher than AMR-NB's FT=8 — and a NO_DATA (DTX-skip) marker at FT=15. AudioToolbox exposes AMR-WB through `kAudioFormatAMR_WB` (`'sawb'`) as a **decode-only** target (same asymmetry as AMR-NB — no paired AT encoder), so the registry only installs an `amr_wb` decoder factory. Storage-format packets per RFC 4867 §5.3: one **TOC byte** (`F FFFF Q PP`, layout shared with AMR-NB), followed by the per-mode compressed payload — 1, 6, 17, 23, 32, 36, 40, 46, 50, 58, or 60 total bytes including the TOC (the speech-mode body bit counts are 132/177/253/285/317/365/397/461/477 per 3GPP TS 26.201 Table 2, so `ceil(bits/8) + 1` for the storage byte count). The decoder validates each incoming packet against the TOC-derived size table before queueing, rejecting reserved frame types (FT=10..=14 for AMR-WB) and size mismatches with `Error::invalid`. Persistent input-queue + one-packet-of-slack lookahead pattern (same as iLBC / AMR-NB / HE-AAC) so AT never sees "0 packets" mid-stream. Tags claimed: FourCC `'sawb'` (3GPP / ISOBMFF) + Matroska `A_AMR/WB`. No magic cookie — the per-packet TOC byte carries everything AT needs.

Round 10 FLAC (Free Lossless Audio Codec, RFC 9639) **decode** via `AudioConverter`. AudioToolbox exposes `kAudioFormatFLAC` (`'flac'`) on macOS 13+ as both a decompression and a compression target; this round installs the decode side, encoder ships in a paired follow-up. The bridge sources its constants from Apple's public `CoreAudioBaseTypes.h` (FourCC `'flac'`, "the flags indicate the bit depth of the source material" → reuses ALAC's four `kAppleLosslessFormatFlag_*BitSourceData` values for `format_flags`). A wall-isolated `src/flac.rs` module parses RFC 9639 STREAMINFO (34-byte big-endian wire form per §8.1) plus the per-frame header (15-bit `0b111111111111100` sync code + blocking-strategy bit + block-size code per Table 1 + sample-rate code per Table 2 + channel-assignment per §9.1.3 + bits-per-sample code per §9.1.4). **Magic-cookie discovery**: AT requires the cookie to be a Xiph "FLAC in ISOBMFF" `dfLa` box (8-byte BoxHeader `[size, 'dfLa']` + 4-byte FullBox `[version=0, flags=0]` + metadata block chain) — discovered empirically by probing `AudioConverterSetProperty(kAudioConverterDecompressionMagicCookie, …)` against every plausible layout (bare STREAMINFO body, `fLaC + STREAMINFO`, multi-block prefixes, etc.); only the `dfLa`-boxed form validates. AT enforces a 256-byte cookie ceiling. Bundled fixture (`tests/fixtures/flac-mono-16bit-44100/`, 10 × 4608-sample fixed-blocksize frames at 44.1 kHz mono 16-bit) decodes to **44 100 i16 samples that match the staged reference WAV byte-exact** — proving the bridge is truly lossless end-to-end. Persistent input-queue + one-packet-of-slack lookahead pattern (same as iLBC / AMR-NB / AMR-WB / MP3) so AT never sees "0 packets" mid-stream. Tags claimed: FourCC `'flac'` (AT / ISOBMFF) and Matroska `A_FLAC`.

Round 9 MP3 (MPEG-1 / MPEG-2 / MPEG-2.5 Audio Layer III, ISO/IEC 11172-3 + 13818-3 + Fraunhofer MPEG-2.5 extension per `docs/audio/mp3/MPEG-2.5-GAP.md`) **decode**: the full Layer-III sample-rate matrix (8 / 11.025 / 12 / 16 / 22.05 / 24 / 32 / 44.1 / 48 kHz) at mono or stereo. AudioToolbox exposes MP3 through `kAudioFormatMPEGLayer3` (`'.mp3'`) as a **decode-only** target — AT ships no MPEG-audio encoder — so the registry installs only an `mp3` decoder factory. There is no magic cookie: the per-frame 4-byte header (sync word + version-id + layer + protection + bitrate index + sample-rate index + padding bit + channel mode) carries everything AT needs. The bridge ships a wall-isolated header parser (`src/mp3.rs`) covering all three version × layer rows and both ISO bitrate tables, computing on-wire frame length per ISO/IEC 11172-3 §2.4.3.1 (Layer III: `144 * br / sr + padding` on MPEG-1, `72 * br / sr + padding` on MPEG-2 / MPEG-2.5). The decoder lazily constructs its `AudioConverterRef` from the first frame header so caller-supplied `CodecParameters` are advisory; subsequent frames must match the latched (version × layer × sample-rate × channel-mode) but bitrate changes are accepted (VBR is in scope). Mid-stream switches surface typed `Error::unsupported` rather than a `kAudioConverterErr_FormatNotSupported`. Persistent input-queue + one-packet-of-slack lookahead pattern (same as iLBC / AMR-NB / AMR-WB) so AT never sees "0 packets" mid-stream. Bundled fixture (`tests/fixtures/mp3-layer3-stereo-44100-128kbps/`, 33 × 1152-sample MPEG-1 LIII 128 kbit/s stereo frames including the leading Xing/Info header frame) decodes to **76 032 interleaved i16 samples** and matches the staged reference WAV at **≈ 89.8 dB SNR** after priming alignment — two independent decoders agreeing within a fraction of a dB across the full 0.85 s window. Tags claimed: FourCC `'.mp3'` (AT / ISOBMFF), MP4 OTI `0x6B` (ISO/IEC 14496-1 MPEG-1 Audio), Matroska `A_MPEG/L3`, and WAVE format tag `0x0055` (Microsoft `WAVE_FORMAT_MPEGLAYER3`).

Round 6 iLBC (Internet Low Bitrate Codec, RFC 3951) decode + encode: a narrow-band telephony speech codec, fixed at **8 kHz mono**. Two block sizes — 20 ms (160 PCM frames → 38 compressed bytes, 15.2 kbit/s) and 30 ms (240 PCM frames → 50 compressed bytes, 13.33 kbit/s) — selected via `CodecParameters::options.insert("mode", "20" | "30")`, defaulting to 30 ms (the compression-favoured mode used by most SIP / RTP gateways). AudioConverter (`kAudioFormatiLBC` = `'ilbc'`) wires the mode purely through the compressed-side ASBD `frames_per_packet` field — there is **no magic cookie**, no in-band signalling, no probe descriptor. Round-trip on a 2-second 1 kHz sine at 8 kHz mono yields **peak SNR ≈ 10.7 dB (30 ms)** and **≈ 7.8 dB (20 ms)**; iLBC is a CELP-class voice codec, so a pure sine is intentionally adversarial (the codebook is voice-tuned) and these SNRs prove the pipeline is wired correctly without claiming transparency, which iLBC doesn't target. Decoder uses a persistent input-packet queue (the same one-packet-of-slack lookahead pattern as the HE-AAC decoder) because AT's iLBC analysis filter draws samples from one or two prior compressed packets when producing a PCM block; without the slack the input callback would return zero mid-stream and AT would silently halt. Encoder echoes the active mode through `output_params.options["mode"]` and the net bitrate (15200 or 13333) through `output_params.bit_rate` for downstream muxer awareness.

Round 218 FLAC **encoder** via `AudioConverter`. Completes the symmetric AT FLAC bridge: encoder packetises interleaved S16 or S32 PCM at 48 kHz / 2-channel through `kAudioFormatFLAC` (`'flac'`) using the same source-data flag convention ALAC pioneered (`format_flags = K_AF_APPLE_LOSSLESS_*BitSourceData` per Apple's `CoreAudioBaseTypes.h` enum comment). **Empirical AT-side discovery (r218 probe)**: the AT FLAC encoder slot rejects S32 → FLAC(32) combos with `kAudioConverterErr_FormatNotSupported` (`'fmt?'` = 1718449215) — AT ships no 32-bit FLAC compression tier on this macOS version, but S16 → FLAC(16) and S32 → FLAC(24) both succeed. The encoder therefore caps the output bit depth at 24 for S32 input (lossless within the 24-bit range, which S32 PCM round-trips byte-identically through the 24-bit compressor). **Second empirical discovery**: AT's FLAC encoder calls back **twice** per `FillComplexBuffer` invocation (one frame-of-input request + one look-ahead pull) and the converter **locks itself at end-of-stream** the moment the PCM callback returns 0 bytes — this poison makes one-shot chunk-at-a-time encoding fail after exactly one packet. The fix is a **persistent PCM feeder context** held in a `Box<PcmContext>` on the encoder, with a one-packet-of-slack discipline (the encoder only pumps FCB once the queue holds at least two packets' worth so the look-ahead pull is always satisfied) — same shape as the iLBC / AMR-NB / HE-AAC slack patterns we developed for the decoder side, applied to the compressed-output side. The encoder vends the resulting `dfLa` magic cookie via `output_params.extradata` for downstream muxer use (or, if the `kAudioConverterCompressionMagicCookie` query fails on an older macOS slot, synthesises a minimal `dfLa` cookie from the configured `(sample_rate, channels, output_bit_depth)` using the same `flac::build_magic_cookie` builder the decoder side validated round 10). Round-trip on a 2-second 1 kHz sine + 12-bit LCG noise at 48 kHz / 16-bit stereo → encode (23 raw FLAC packets, 6 KB / packet) → decode → **188,416 / 192,000 i16 samples bit-exact at offset 0** (zero priming silence on the AT FLAC encoder path — same lossless contract ALAC ships). Three lifetimes of registration work: r10 decode, r218 encode, future rounds will explore fuzz / benchmark coverage now that the surface is symmetric.

Round 224 Opus (IETF RFC 6716 + RFC 7845 + RFC 8251) **decode + encode** via `AudioConverter`. AudioToolbox exposes `kAudioFormatOpus` (`'opus'`, per Apple's public `CoreAudioBaseTypes.h`: "Opus codec, has no flags") as a symmetric decompression + compression target. The bridge ships a wall-isolated `src/opus.rs` module that parses and builds the RFC 7845 §5.1 ID header ("OpusHead"): 8-byte magic + 1-byte version + 1-byte channel count + 16-bit pre-skip + 32-bit input sample rate + 16-bit Q7.8 output gain + 8-bit mapping family + optional mapping-table tail, all multi-byte fields little-endian. Mapping family 0 (the RTP 1-or-2-channel layout per RFC 7587) serialises to a 19-byte cookie; families 1 and 255 carry the 2-byte stream/coupled prefix plus one byte per output channel. **Empirical AT-side discovery (r224 probe)**: the Opus decoder slot **locks the converter at end-of-stream the moment the input callback returns `0 packets`** — a single empty-queue callback is enough to enter the lockout state, after which subsequent `FillComplexBuffer` calls return 0 PCM frames regardless of further input. The fix is a strict "drain only while the queue holds at least 2 packets" discipline plus bounding the per-call PCM output to one packet's worth of frames (`io_number_data_packets = sample_rate * 20 / 1000`) so AT consumes one Opus packet per FCB and the slack invariant always holds. Decoder output rates per RFC 6716 §2.1.1: 8 / 12 / 16 / 24 / 48 kHz; the bridge defaults to 48 kHz (the RFC 7845 §5.1 recommended playback rate). Encoder accepts S16 or F32 PCM at any of those rates, plumbs `CodecParameters::bit_rate` (default 96 000 bit/s) through `kAudioConverterEncodeBitRate` and reads back the AT-quantised value, and exposes the per-packet duration via `options.insert("frame_duration_ms", …)` over the six RFC 6716 Table 2 values (2.5 / 5 / 10 / 20 / 40 / 60 ms → 120 / 240 / 480 / 960 / 1920 / 2880 PCM frames at 48 kHz). The encoder publishes an RFC 7845 OpusHead through `output_params.extradata` for downstream Ogg / MP4 / WebM muxer use. Round-trip on a 2-second 1 kHz sine at 48 kHz / stereo → encode (100 raw packets at ≈ 96 kbit/s) → decode → **≈ 191 760 / 192 000 PCM samples recovered, peak SNR ≈ 26.1 dB per channel** after pre-skip alignment search; a mono companion case exercises the 1-channel mapping-family-0 path at ≈ 26.4 dB. Opus is a perceptual lossy codec (RFC 6716 §1) so a pure sine is intentionally adversarial — the SNR floor is set at 6 dB for the same reason iLBC's is; the goal is wired-pipeline verification, not transparency. Tags claimed: FourCC `'Opus'` (ISO/IEC 14496-12 Opus sample-entry FourCC) and Matroska `A_OPUS`.

Round 212 ALAC **S32 output path** (decoder tightening): before this round `AlacAtDecoder` hard-coded its output ASBD to `pcm_s16` regardless of the cookie's `bit_depth`, so every 24- or 32-bit ALAC track silently lost its low-order bits even though the encoder side correctly emitted a lossless bitstream — defeating the codec's lossless contract on its native bit depths. A new `pcm_s32` ASBD constructor in `sys.rs` (signed-integer + packed, 32-bit, distinct from the existing `pcm_float32`) wires through to a per-instance `output_format` on `AlacAtDecoder`: callers asking for `CodecParameters::sample_format = Some(SampleFormat::S32)` now receive the full 32-bit word; `None` / `S16` keeps the legacy S16 path so every existing caller is byte-identical. Asking for S32 against a 16- or 20-bit cookie is accepted (AudioConverter sign-extends the source word into the high bytes). A new `tests/alac_s32_roundtrip.rs` exercises the full encode → decode loop at S32 with a 440 Hz sine plus a deterministic **24-bit low-bit pseudo-random noise term** that sits entirely below the S16 quantisation floor — a decoder that truncates to S16 destroys the noise term and fails the bit-exact prefix assertion, so the test functions as a regression seal. Result: **190,464 / 192,000 i32 samples bit-exact** end-to-end, matching the S16 round-trip count exactly. Three unit tests round it out (`default_output_is_s16`, `explicit_s32_switches_output_width`, `s32_with_24bit_cookie_accepted`) and `AlacAtDecoder::output_sample_format()` lets callers introspect which width they'll receive.

Round 265 typed `AudioFormatId` introspector (sys-surface tightening): the CoreAudio C surface treats `AudioStreamBasicDescription::format_id` as a raw `u32` FourCC (`kAudioFormat*`). That works for setting fields but loses type-system traction the moment a caller wants to introspect "which codec is this ASBD describing" — every introspection collapses to a match ladder against the bare integer constants. This round adds a typed `AudioFormatId` enum with one variant per wired `K_AUDIO_FORMAT_*` constant (15 named arms: linear-PCM + 14 compressed codec slots covering AAC-LC / AAC-HE / AAC-HE-v2 / AAC-LD / AAC-ELD / ALAC / iLBC / AMR-NB / AMR-WB / MPEG Layer I / II / III / FLAC / Opus) plus an `Unknown(u32)` tail-case that preserves the raw value for unwired FourCCs so the classifier stays total. `AudioFormatId::from_u32` / `as_u32` provide a round-trip pair; `is_linear_pcm()` / `is_compressed_audio()` typed predicates split the wired set on the only axis CoreAudio FFI consumers actually branch on. Three new accessors on `AudioStreamBasicDescription` (`audio_format_id()`, `is_linear_pcm()`, `is_compressed_audio()`) forward to the typed surface so call sites that already hold an ASBD never need to re-read the raw u32. The primitive is purely additive — every existing constructor and FFI path is byte-identical — and 5 new unit tests pin the round-trip (every wired constant classifies to its named variant + back), the Unknown tail-case (a `'zzzz'` FourCC stays as `Unknown(0x7a7a7a7a)`), the predicate-axes invariant (every wired variant satisfies `is_linear_pcm() XOR is_compressed_audio()`), the ASBD-constructor / typed-variant correspondence across all 17 wired constructors, and a hand-crafted unwired-FourCC ASBD reporting `false` on both predicates. Adding the next codec slot is now a four-line change: a `K_AUDIO_FORMAT_*` const, an enum variant, and two arms in `from_u32` / `as_u32`.

## Opt-out

Disable hardware acceleration globally via `CodecPreferences { no_hardware: true, .. }` or the `oxideav` CLI's `--no-hwaccel` flag. The runtime context still registers AT — `oxideav list` shows the `aac_audiotoolbox` row regardless of the flag — only resolution is biased toward the SW path.

## Workspace policy

Calling a system OS framework via FFI is the same shape as calling `libc::malloc` — it's the platform, not a copied algorithm. The workspace's clean-room rule does not apply to this crate.
